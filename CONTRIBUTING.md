# Contributing to pdfkit

Thanks for your interest in contributing! pdfkit is a from-scratch, AI-oriented
PDF toolkit in Rust. This guide covers how to get set up, the invariants the
codebase holds, and how to get a change merged.

By participating you agree to keep the discussion respectful and constructive.

## Getting started

See the [README](./README.md#setup) for full setup. The default build is
intentionally small — Rust only, no native deps, no network after Cargo resolves:

```bash
git clone git@github.com:anshu-gautam/pdf-kit.git
cd pdf-kit
rustup show            # rust-toolchain.toml selects the channel + components
cargo build --workspace
cargo test --workspace
```

The MSRV is **Rust 1.82.0** (enforced in CI). PDFIUM, OCR models, the HTTP API,
and the web UI have their own optional setup steps in the README.

## Architecture invariants (please don't break these)

These are the rules that keep the project portable, deterministic, and safe.
A change that violates one will not be merged without discussion.

1. **Deterministic, offline core.** No network calls and no hosted-model calls
   anywhere in the default build. The only place a network model is touched is
   `pdfkit-adapters` behind the `llm-adapter` feature, where the *caller* supplies
   the client. Local ML for OCR (ONNX via `ocrs`) is fine.
2. **Pure-Rust by default.** `cargo build`/`cargo test` and the minimal build
   must compile with **zero native dependencies and zero network access**. Any
   heavy, native, or networked dependency goes behind a Cargo feature — never in
   the default set. (PDFIUM, Tesseract, and the axum HTTP stack are all gated or
   isolated this way.)
3. **Reading and writing are separate.** `pdfkit-edit` depends only on
   `pdfkit-core` and never flows through the extraction engine.
4. **One-based page numbers** in all public API; convert at the boundary.
5. **No panics on library paths.** Library code returns `Result<_, PdfError>` —
   no `unwrap()`, `expect()`, or `panic!`. PDFs are hostile input; a `cargo-fuzz`
   harness plus an always-on no-panic test guard this.
6. **The `pdfkit-api` crate is a leaf binary** excluded from `default-members`;
   its async/network stack must never reach the library build or the minimal CI
   gate.

Adding a dependency? Keep it MIT/Apache-2.0 compatible. **Flag any GPL-licensed
dependency before pulling it in** — it would force the whole project's license.

## Development workflow

1. Branch off `main` (e.g. `feat/...`, `fix/...`, `docs/...`).
2. Make focused changes; prefer test-first where practical.
3. Run the required checks below — all must be green before you push.
4. Open a PR against `main` with a clear description of the deliverable.

### Required checks

```bash
cargo fmt --all
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
cargo test --workspace --no-default-features --features render-native   # zero-native-dep build
```

If you touched the HTTP API (it's excluded from the `--workspace` gates above):

```bash
cargo clippy -p pdfkit-api --all-targets -- -D warnings
cargo test  -p pdfkit-api
```

If you touched the web UI (`apps/web`, needs Node 20.9+):

```bash
cd apps/web && npm run lint && npm run build
```

Optional / backend checks (run when relevant):

```bash
wasm-pack build crates/pdfkit-wasm
cargo build -p pdfkit-render -p pdfkit-cli --features render-pdfium
cargo test  -p pdfkit-ocr --features ocr-ocrs
cargo +nightly fuzz run --target x86_64-unknown-linux-gnu parse        # deep fuzzing
```

CI (`.github/workflows/ci.yml`) runs fmt, clippy, default + minimal tests, the
wasm build, MSRV, the `api-build` job, and a fuzz smoke run. A PR merges only
when CI is green.

## Commits & pull requests

- Keep commits small and focused; one logical change per commit.
- Use clear, conventional-style messages (`feat:`, `fix:`, `docs:`, `core:`,
  `chore:` …) — match the existing history.
- Describe **what** the change delivers and **why** in the PR body; link any
  issue. Note if you added a feature flag, dependency, or fixture.
- Don't vendor OCR models or large binaries into git — they download into a
  cache dir via the `scripts/fetch-*.sh` helpers.

## Tests & fixtures

- Unit tests live next to the code; integration tests in each crate's `tests/`.
- Deterministic synthetic PDFs live in `fixtures/` and are generated by
  `cargo run -p pdfkit-fixtures --bin write-fixtures` — add a fixture there rather
  than committing a large or copyrighted real-world PDF.
